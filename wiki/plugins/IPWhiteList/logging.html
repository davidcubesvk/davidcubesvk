Basic information:

IPWhiteList has logging system built-in for easier troubleshooting from both user's and developer's side. Customization is available in IPWhiteList's config.yml file. It is heavily recommended to have logs enabled.

Log files:

Each log file represents one lifecycle of the plugin - from time when it's enabled till it's disabled. All log files are saved in folder named ,,logs" located in plugin's folder and are saved in format yyyy-MM-dd_HH:mm:ss.log represeting date and time when was the log file created. All entries in log file are saved in format [yyyy-MM-dd HH:mm:ss] LEVEL: [PREFIX] message, please continue reading for all levels and prefixes.

Levels:

Information in logging files are logged on 3 different levels:

Level Description
INFO Messages informing about statuses or processes in the plugin.
WARNING Messages related to wrong configuration or invalid values.
ERROR Messages related to errors (example: ,,Error while connecting to the database server!").

Prefixes:

Every logged message has it's own prefix representing the source of the message:

Prefix (source) Description
- General message (example: ,,Registering listeners and commands.").
YML Messages related to YML connection type.
SOCKET Messages related to SOCKET connection type.
MYSQL Messages related to MYSQL connection type.
CONNECTOR Messages informing if player is allowed to join, or if not, displays the cause.
UPDATER Updater messages.
API Messages related to plugin's API (example: ,,Getting BungeeCord's port.").

Reading the logs:

Very important messages about connection status, or new update messages are also printed into the console, but in longer format. In logs messages are shorter because there are prefixes, which specify the source of the message. Example:

Console: 14:01:51 [INFO] [IPWhiteList] ServerSocket started on port 4444.

Log: [2020-03-05 14:01:51] INFO: [SOCKET] Started on port 4444.

Console logging also brings one disadvantage, there aren't logged the full error stacktraces, only message describing that some error has occurred and in which operation. In this case you should open the latest log locate the message and look at the stacktrace message. If it is something that you are not familiar with and can't fix it, please send the error to the developer.

Other case when you should look into the logs is when you think something doesn't work or on the other side, make sure if everything works properly, for example, whitelisted IPs synchronization messages.
