1. Installation:

Download the latest version of the plugin available from the official plugin page on SpigotMC. Put the .jar file into the plugins folder of your BungeeCord server and restart it. Next steps will guide you through the configuration itself.

2. Connecting to IPWhiteListBukkit:

config.yml

#Connection with extension IPWhiteListBukkit:
connect:
  #Type of the connection (more and detailed info can be found on the wiki):
  #- YML: connection provided by data.yml file created in plugin's data folder
  #- SOCKET: connection using sockets
  #- MYSQL: connection using MySQL database
  type: SOCKET
  ...

IPWhiteListBukkit (IPWhiteList's extension) is a requirement that needs to be installed on your every Spigot server. These two plugins need to communicate between each other, we are going to set up the connection. Please locate section connect: in configuration file.

type: This specifies the connection type that will be used. We have three options.

  • YML - Plugins communicate using data.yml file generated in IPWhiteList's plugin folder. Please see the scheme below to see how the communication works. Skip step 3 as there is no configuration for this connection type on IPWhiteList's side.

  • SOCKET - Synonym to communication provided over internet. Plugins communicate ,,directly" by sending messages to each other whenever needed. This connection consists of a server and clients. The server is IPWhiteList and clients are IPWhiteListBukkit plugins. Imagine you are opening a website, the website is a server and your browser is a client. This is also the same when you connect any game server - you are client.

  • MYSQL - Recommended. Plugins communicate using a database. Please see the scheme below to see how the communication works.

Comparision:

Here are hosting types supported by each connection type, their pros and cons:

Connection type Supports Pros (+) Cons (-)
YML All servers except shared hosting. Reliable. Slow.
SOCKET All servers (hosting types). Fast. Unsecure (does not use SSL), hard to set up.
MYSQL All servers (hosting types). Fast, reliable, can use SSL, easy and familiar to set up, database is standard for many servers. -

3. Connection configuration:

3A. Socket configuration:

config.yml

#Connection with extension IPWhiteListBukkit:
connect:
  ...
  #Socket port (if type = SOCKET):
  socketPort: 4444
  ...

If we chose SOCKET connection type, we need to configure only one thing. Please keep in mind you must open this port on your machine (if you are not using local connection - e.g. you do not have all servers on one machine):

socketPort: Port on which IPWhiteList should host it's socket server. This is not the port of your server.

3B. MySQL configuration:

config.yml

#Connection with extension IPWhiteListBukkit:
connect:
  ...
  #MySQL connection (if type = MYSQL):
  mysql:
    #Host:
    host: ""
    #Port:
    port: ""
    #Username
    username: ""
    #Password:
    password: ""
    #Database:
    database: ""
    #Table name to use:
    table: "ipwhitelist"
    #Use SSL certificate to connect (secure way for communication):
    useSSL: true
    #Delay between reconnecting attempts, when connection has been unexpectedly closed (in seconds):
    reconnect: 3
    ...

If we chose MYSQL connection type, we need to configure everything needed for MySQL connection.

host: IP of the database host.

port: Port on which is the database hosted (usually 3306).

username: Username to connect with to the database.

password: Password for the username.

database: Name of the database to use.

table: Table to be used by IPWhiteList. Do not create the specified table! IPWhiteList will create it by itself with needed columns.

useSSL: Use SSL certificate to connect.

reconnect: Delay in seconds between reconnecting attempts, when connection unexpectedly failed (or initially failed to connect) and trying to connect again. Set to 0 for instant reconnecting.

4. Whitelisted IPs:

config.yml

#WhiteListed IPs:
#INFO: 1. To enable all hostnames, insert *. prefix before the whitelisted IP, e.g. *.yourserver.com will allow players connecting with
#         abc.yourserver.com, xyz.yourserver.com..., (useful for servers with AntiDDoS service).
#      2. To apply whitelisted IP without hostnames, insert only the IP without any prefix, e.g. yourserver.com will allow only players
#         connecting with yourserver.com.
#      3. If your server has dynamic IP, use {ip} placeholder as the server's IP (useful for dynamic IP servers).
#      4. To allow not case-sensitive hostname comparing for the current IP, insert "[NOT_CASE_SENSITIVE]" or "[CASE_SENSITIVE]"
#         (for case-sensitive comparing) right before the IP as shown in example whitelisted IPs list. If not inserted, the default
#         setting is case-sensitive.
#NOTE: You don't need to write ports there. IPWhiteList will automatically obtain port where is BungeeCord server hosted on.
#      However, if you write port there, BungeeCord port will be overridden by specified port.
whitelisted:
  - 'yourserver.com'
  - 'localhost'
  - '[NOT_CASE_SENSITIVE]abc.yourserver.com'

Now we need to specify the IPs we want players to connect with. Whitelisted IPs can be added to the list under whitelisted: tag. Each player connecting with any of these IPs will be allowed to connect, others will get disconnected during the logging in process. You can whitelist number-IPs, domains, hostnames and ports.

  • Whitelisting ports - Easily, just specify the IP and the wanted port separated with a colon, for example yourserver.com:25565 will allow players connecting only with this IP and port. If not specified, IPWhiteList will obtain and use the port on which is the BungeeCord server hosted. For example, you have BungeeCord hosted on port 25565 and you have whitelisted yourserver.com, player connecting with the port 25565 and this IP will be allowed (with yourserver.com:25565).

  • Whitelisting hostnames - Hostnames can be whitelisted easily by adding them in front of the IP, for example mc.yourserver.com will allow players connecting with mc.yourserver.com (see the upper point if you'd like to know how ports work). If you want to allow all hostnames, use *. wildcard as hostname before the IP; this means *.yourserver.com will allow players connecting with abc.yourserver.com, xyz.yourserver.com etc., but it will not allow players connecting with yourserver.com (without any hostname).

  • For servers with dynamic IP address - If the machine on which is your BungeeCord server hosted has dynamic IP (it's IP changes every server restart), just use placeholder {ip} as a replacement for the machine's IP. As it is just a replacement, you can specify ports, hostnames, without any problems.

  • IP comparing prefixes - IPs (the one used by player and a whitelisted IP) are compared case-sensitively by default. To change the coparision mode, insert [CASE_SENSITIVE] or [NOT_CASE_SENSITIVE] prefix before the IP. For example, [NOT_CASE_SENSITIVE]abc.yourserver.com will also allow players connecting with aBc.yoUrSerVer.com.

5. MySQL synchronization:

config.yml

#Connection with extension IPWhiteListBukkit:
connect:
  ...
  #MySQL connection (if type = MYSQL):
  mysql:
    ...
    #Whitelisted IPs synchronization:
    #INFO! Very useful for large networks with more than one proxy. In this case you don't need to copy whitelisted IPs
    #      anytime you change them to every config, just synchronize whitelisted IPs (all IPWhiteList plugins) with one
    #      central IPWhiteList.
    whitelistedSync:
      #Synchronization mode:
      #- NONE: do not query / upload whitelisted IPs to the database and use the config.yml file for them
      #- HOST: host the whitelisted IPs (central IPWhiteList) - upload whitelisted IPs from config.yml file to the database
      #        so other IPWhiteList plugins set on SYNC mode can query them and synchronize them
      #        WARNING! If changes in whitelisted IPs were made between restarts, central IPWhiteList's server must start as first.
      #- SYNC: synchronize whitelisted IPs with the central IPWhiteList by querying the database
      mode: NONE
      #Table name to store whitelisted IPs in:
      table: "ipwhitelist_ips"

This feature is available only for MYSQL connection type and is meaningful only if you have more than one proxy. This feature synchronizes whitelisted IPs across all your proxies.

mode: This specifies in which mode - which role current IPWhiteList plugin has in the synchronization process. We have three options. The synchronization process must have only one central IPWhiteList set on mode HOST, the rest of IPWhiteList plugins must be set on mode SYNC.

  • NONE - No synchronization, using whitelisted IPs saved in config.yml.

  • HOST - Host of the synchronization process (central IPWhiteList), upload whitelisted IPs from config.yml to the database, so other IPWhiteList plugins can synchronize with these whitelisted IPs. If changes in whitelisted IPs (in central (HOST) IPWhiteList) have been made please restart central IPWhiteList's server and then other proxies.

  • SYNC - Synchronize whitelisted IPs with the central IPWhiteList, get whitelisted IPs from the database and save them into plugin's config.yml file.

table: Table to be used by IPWhiteList for whitelisted IPs synchronization. Do not create the specified table! IPWhiteList will create it by itself with needed columns.

6. Setup mode:

config.yml

#Setup mode (toggleable using /ipwl setup):
#(set this to false when you set up whitelisted IPs, to make them working. If true, plugin ignores whitelisted IPs)
setup: true

Finally, we need to turn the setup mode off, so IPWhiteList can protect your server. The setup mode can be toggled by setting setup: to true (on) or false (off), or using /ipwl setup command. If on, plugin will accept all connections not depending on if they are connecting with a whitelisted IP or not.
